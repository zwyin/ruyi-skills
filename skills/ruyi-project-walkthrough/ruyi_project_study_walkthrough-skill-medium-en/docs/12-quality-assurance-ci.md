# Quality Assurance & CI

The project enforces quality through automated verification at multiple levels: converter-level checks, manifest validation, and CI pipeline integration.

## Quality Gates Overview

Quality enforcement happens at three points in the pipeline:

```
Phase 3C: Manifest validation (verify_sources.py)
    ↓
Phase 4:  Converter verification (md_to_html.py --verify)
    ↓
Phase 5:  Delivery gate (verify-result.json + spot checks)
```

No output reaches the user without passing all three.

## Phase 5: Delivery Gate

Phase 5 is the final checkpoint with four steps:

### Step 1: Automated Structural Verification

The converter automatically runs verification after HTML generation and writes `verify-result.json`:

```json
// Simplified from: scripts/md_to_html.py:527-538
result = {
    "passed": len(errors) == 0,
    "timestamp": datetime.now(timezone.utc).isoformat(),
    "html_file": str(html_path),
    "html_size": file_size,
    "markdown_size": total_md_size,
    "section_count": section_count,
    "expected_sections": expected_chapters,
    "ratio_pct": round(ratio_val * 100) if ratio_val else None,
    "sidebar_nav_count": sidebar_links,
    "errors": errors,
}
```

Six verification checks:

| Check | What It Validates |
|-------|------------------|
| Section count | HTML chapters = .md file count |
| Per-section content | Each chapter has heading + content element |
| File size ratio | HTML >= 80% of markdown total |
| No broken .md links | Zero `href="*.md"` in HTML |
| Navigation integrity | Sidebar items = chapter count |
| Quiz presence | Quiz section exists if `--quiz-chapter` was used |

### Step 2: Automated Content Verification

Additional programmatic checks:

```bash
# No .md href links
grep -c 'href="[^"]*\.md"' <output.html>     # must be 0

# Count code blocks vs manifest entries
grep -c '<pre><code>' <output.html>

# File count in docs/ matches expectations
ls <docs_dir>/*.md | wc -l
```

### Step 3: Error Recovery Loop

If verification fails, the system attempts up to 3 automated fixes:

```text
// Simplified from: skills/ruyi-project-walkthrough/SKILL.md:709-719
| Error pattern | Action |
|--------------|--------|
| "Section count X != expected Y" | Re-generate missing .md files |
| "Chapter N is empty" | Re-read .md file, re-generate if empty |
| "HTML size ... below 80%" | Compare sections to identify dropped content |
| "Found N broken .md links" | Re-run converter (safety net strips these) |
| "Sidebar nav X < chapter count Y" | Re-run converter |
```

After each fix attempt, verification is re-run. If 3 retries are exhausted, markdown is delivered without HTML, with specific error messages and the manual converter command.

### Step 4: Final Delivery Gate

The delivery gate requires:

**Automated checks:**
- `verify-result.json` exists alongside HTML
- `verify-result.json` contains `"passed": true`
- No broken `.md` links in HTML
- `sources-manifest.json` exists for each depth level
- Every code block has a manifest entry

**Model-verified checks:**
- 2-3 chapters spot-checked for content completeness
- No placeholder or TODO content
- Quiz answers are correct

## CI Pipeline

GitHub Actions runs on push to master/develop and on PRs:

```makefile
// Simplified from: Makefile:18-19
ci: test check
	@echo "CI checks passed."
```

Two CI jobs:

1. **`test`** — `pytest tests/ -v` runs all 8 test files
2. **`check`** — `./scripts/convert.sh --check` verifies platform files are in sync

Both must pass for CI to succeed. The `release.sh` script waits for CI to pass before merging release PRs.

## Quality Metrics Report

The final delivery report follows a structured template:

```text
Walkthrough Generation Complete
================================
Output: <output_directory>
├─ Markdown: <N> chapters in docs/
├─ HTML: walkthrough-<project>-<depth>-<timestamp>.html
├─ Manifest: sources-manifest.json (<M> verified claims)
├─ Verification: verify-result.json
└─ Verified at: <timestamp>

Quality metrics (from verify-result.json):
├─ Sections: <count> (expected: <expected>)
├─ HTML size: <size>Kb (markdown: <size>Kb, ratio: <%>%)
├─ Nav items: <count>
├─ Code blocks: <C> (manifest entries: <M>)
└─ Status: PASSED ✓
```

All metrics must be read from `verify-result.json` — fabrication of values is explicitly forbidden.

## verify-result.json as Evidence

This file is the single source of truth for quality evidence:

- It is auto-generated by the converter
- It contains structured, machine-readable metrics
- It cannot be claimed as PASSED without the file existing
- It includes a UTC timestamp for audit purposes

If the file does not exist, the report must say "Verification: NOT RUN" — not claim PASSED.
