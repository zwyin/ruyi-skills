# 02 五阶段 Pipeline 架构

## 整体流程

project-walkthrough 的核心是一个 5 阶段 pipeline，从分析到交付，每阶段有明确的输入、输出和质量门禁。

> 类比理解：这个 pipeline 就像一条工厂流水线——原材料（源码）进来，经过 5 道工序（分析→探索→规划→生成→验证），最终产出成品（markdown + HTML）。每道工序都有自己的质检标准，上一道不合格，下一道不接收。

## Pipeline 数据流

```
用户输入 (--depth, --audience, --path)
    │
    ▼
┌─────────────────────────┐
│  Phase 0: 分析与确认     │  ~5% 时间
│  ├─ 0.1 加载偏好         │
│  ├─ 0.2 内容分析         │  → analysis.md
│  └─ 0.3 确认门           │  → 用户确认 scope/depth/format
└─────────┬───────────────┘
          │
          ▼
┌─────────────────────────┐
│  Phase 1: 探索           │  ~15% 时间
│  ├─ 识别项目类型          │
│  ├─ 读核心文件           │
│  ├─ 读关键源码           │
│  ├─ 提取创新点           │
│  └─ 映射架构             │
└─────────┬───────────────┘
          │
          ▼
┌─────────────────────────┐
│  Phase 2: 规划章节        │  ~10% 时间
│  ├─ 选择章节模板          │
│  ├─ 确定章节数量          │
│  ├─ 适配项目特征          │
│  └─ 映射源文件到章节      │
└─────────┬───────────────┘
          │
          ▼
┌─────────────────────────────────────────┐
│  Phase 3: 生成 Markdown                  │  ~40% 时间
│  ├─ 3A 验证声明 → sources-manifest.json │  ← 写入许可证
│  ├─ 3B 写入章节内容（仅已验证声明）      │
│  └─ 3C 校验覆盖率                        │
└─────────┬───────────────────────────────┘
          │
          ▼
┌─────────────────────────┐
│  Phase 4: 生成 HTML      │  ~30% 时间
│  ├─ 运行 md_to_html.py   │  ← Converter-first
│  ├─ 生成交互式 HTML       │
│  └─ 自动质量校验          │  → verify-result.json
└─────────┬───────────────┘
          │
          ▼
┌─────────────────────────┐
│  Phase 5: 验证交付       │  ~5% 时间
│  ├─ 文件完整性检查        │
│  ├─ Manifest 覆盖率       │
│  ├─ HTML 有效性           │
│  └─ Quiz 答案正确性       │
└─────────────────────────┘
```

## 各阶段时间分配

| 阶段 | 占比 | 核心产出 |
|------|------|---------|
| Phase 0 | ~5% | analysis.md + 用户确认 |
| Phase 1 | ~15% | 探索笔记（不落盘） |
| Phase 2 | ~10% | 章节大纲 |
| Phase 3 | ~40% | 12 个 .md 文件 + sources-manifest.json |
| Phase 4 | ~30% | 1 个交互式 HTML + verify-result.json |
| Phase 5 | ~5% | 验证报告 |

## Phase 3 的三阶段子流程

Phase 3 是整个 pipeline 的核心，因为它保证准确性。它分为三个子阶段：

### Phase 3A：验证与构建 Manifest

**写入许可机制**——manifest 不是审计日志，而是写入许可证。每个声明必须在 manifest 中有 `verified: true` 记录，才能在 Phase 3B 中被写入。

验证步骤：
1. 列出计划的所有声明（代码示例、目录结构、API 签名、版本号等）
2. 逐个读取源文件验证（不是从 README 或记忆中推断）
3. 创建 manifest 条目记录来源和验证状态
4. 不可验证的声明被丢弃，用可验证的替代

### Phase 3B：写入章节内容

只使用 Phase 3A 中已验证的声明。每个代码块必须有 `// Simplified from: path:lines` 引用。

### Phase 3C：校验覆盖率

回读每个章节，交叉检查所有事实声明是否有 manifest 条目，确保无遗漏。

## 为什么是这个顺序

这个 pipeline 的设计哲学是"验证先行"（verify before write）。传统做法是先写后验，但这会导致：
- LLM 在生成内容时"编造"合理的代码片段
- 事后发现错误时需要大面积重写
- 读者无法判断内容是否可信

通过把验证（Phase 3A）放在写入（Phase 3B）之前，project-walkthrough 在结构层面消除了编造内容的可能性。
