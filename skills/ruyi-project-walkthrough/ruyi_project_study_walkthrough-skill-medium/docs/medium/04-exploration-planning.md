# 04 Phase 1-2：探索与章节规划

## Phase 1：探索协议

探索协议（`docs/exploration-protocol.md`，206 行）定义了如何系统性地了解一个项目。它包含 6 个步骤：

### Step 1：识别项目类型

通过检查顶层文件来分类项目：

| 类型 | 指标 | 典型项目 |
|------|------|---------|
| AI tool | Skills/prompts 目录，AI 模型引用，plugin 系统 | superpowers, cursor |
| Library | `src/index.ts` 导出，`lib/` 目录，API 文档 | React, Lodash, zod |
| Web app | 路由/页面目录，服务器入口，数据库迁移 | Next.js, Rails |
| CLI tool | `bin/` 目录，commander/yargs 依赖 | git, esbuild |
| Game engine | 渲染器/着色器/物理目录，ECS，场景图 | Bevy, Godot |
| Database | 存储引擎，WAL，查询规划器，B-tree/LSM-tree | RocksDB, SQLite |
| Compiler | 词法分析器/解析器，AST，类型检查器，代码生成 | Rustc, TypeScript |
| Document | 连续 markdown 章节，无源码文件 | 研究报告，学习指南 |

### Step 2-3：读核心文件和关键源码

按优先级读取文件：

1. README.md、package.json、CLAUDE.md/AGENTS.md — 所有项目必读
2. 核心逻辑模块 — 做主要工作的代码
3. 配置 — 项目如何配置
4. 入口点 — 执行从哪里开始
5. 测试 — 项目认为值得测什么（揭示优先级）

**文件大小感知策略**：

| 文件大小 | 策略 |
|---------|------|
| < 500 行 | 全文阅读 |
| 500–2,000 行 | 前 100 行 + 关键段落 |
| > 2,000 行 | 先看大纲/符号，只读需要的段落 |

### Step 4：提取创新点

识别 3-5 个独特的设计决策、技术或关键发现。问四个问题：
- 这个项目和其他类似工具有什么不同？
- 哪些设计决策会让资深开发者惊讶？
- 它解决了别人没解决的问题吗？
- 作者做了什么权衡，为什么？

### Step 5：映射架构

构建项目的心理模型：组件、数据流、依赖、配置、状态。

### Step 6：验证探索发现

交叉检查关键声明，标记不可验证的为 `[UNVERIFIED]`。这些声明必须在 Phase 3 中重新验证，否则从 walkthrough 中省略。

## Phase 2：章节规划

章节规划使用 `docs/chapter-templates.md`（401 行）中的模板，按项目类型选择：

| 项目类型 | Brief 章节数 | Medium 章节数 |
|---------|-------------|--------------|
| AI Tool | 7-8 | 12-15 |
| Library | 7-8 | 12-15 |
| Web App | 7-8 | 12-15 |
| CLI Tool | 7-8 | 12-15 |
| Document | 1:1 保留 | 选择关键章节 |
| Game Engine | 7-8 | 12-15 |
| Database | 7-8 | 12-15 |
| Compiler | 7-8 | 12-15 |

模板是起点，不是约束。规划时需要：
1. **删除无关章节** — 如果项目没有安全模型，不硬造一个
2. **添加独特章节** — 如果项目有不寻常的功能，加专门章节
3. **合并重叠章节** — 如果两个模板章节在特定项目中重叠，合并
4. **尊重项目自身组织** — 如果项目有清晰的模块结构，在章节中反映

## 章节质量标准

每个章节应满足：
- **一个清晰主题** — 不是"杂项功能"
- **有源文件支撑** — 可以指向具体文件
- **可独立阅读** — 不需要读其他章节也能理解
- **有实质内容** — 不是"这个文件有 200 行代码"
